# 厨房管理小程序设计文档

- **日期**：2026-06-25
- **状态**：设计已确认，待实现计划
- **场景定位**：家庭/合租厨房（C 类场景）

---

## 1. 项目概述

### 1.1 一句话定位

> **家庭冰箱的智能大脑**——以「圈子」为隔离边界、「库存」为核心枢纽的厨房协作小程序。

### 1.2 核心场景

合租 / 家庭厨房里的供需协作：
- 谁有空谁做（主理人），谁想吃谁提（使用者下单）
- 冰箱里有什么、什么快过期、该买什么——库存是核心
- 一人可属于多个厨房圈子（"我和室友"、"我和爸妈"）

### 1.3 核心痛点与优先级

| 梯队 | 痛点 | MVP 是否解决 |
|---|---|:---:|
| 第一梯队 | ④ 食材浪费/过期 | ✅ |
| 第一梯队 | ⑤ 食材归属（澄清后：实际是"谁垫的钱"，归二期费用模块） | ⚠️ 二期 |
| 第二梯队 | ① 谁做饭的协调 | 二期 |
| 第二梯队 | ② 费用分摊 | 二期 |
| 第二梯队 | ③ 口味/忌口 | 二期 |

> **透明说明**：原始痛点 ⑤（归属）经澄清为"谁垫的钱"，本质属费用范畴，已移至二期。MVP 库存的核心痛点收敛为 **④ 临期/浪费**。

## 2. 架构总览

### 2.1 核心抽象：以「圈子」为隔离边界

所有业务数据挂在 `Kitchen` 下，跨厨房完全隔离。用户通过 `Membership` 加入厨房，角色信息记录在成员关系上（**不在 User 上**），天然支持"一人在不同厨房有不同角色"。

```
User ──< Membership >── Kitchen ──< 业务数据
            (角色在这)
```

### 2.2 三层结构

```
┌─────────────────────────────────────┐
│  厨房/圈子 Kitchen                   │  ← 隔离边界
│  ┌───────────┐  ┌──────────────┐    │
│  │ 成员/角色  │  │ 菜谱库 Recipe │    │
│  └───────────┘  └──────────────┘    │
│  ┌───────────┐  ┌──────────────┐    │
│  │ 菜品菜单   │  │ 库存 Inventory │  ← 核心枢纽
│  └───────────┘  └──────────────┘    │
│  ┌─────────────────────────────┐    │
│  │ 订单 Order(+明细+状态)       │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

### 2.3 技术栈（已确认）

全 TypeScript 技术栈，跨端能力优先。

| 层 | 选型 | 说明 |
|---|---|---|
| **语言** | TypeScript（全栈） | 前后端共享类型，Prisma 类型贯穿到前端 |
| **前端** | uni-app（Vue 3 + `<script setup>`） | 一套代码编译多端（微信小程序 / H5） |
| **后端** | NestJS | 结构化、依赖注入、模块化，利于单人长期维护 |
| **ORM** | Prisma | 类型安全，支持 SQLite/PostgreSQL 无缝切换 |
| **数据库** | 开发期 SQLite / 上线 PostgreSQL | 开发零安装，上线用托管 PG |

**选型理由**：
- 全 TS + Prisma 生成的类型可贯穿到 uni-app 端，避免前后端重复定义类型。
- Prisma 屏蔽 SQLite↔PG 差异，开发期用一个 `.db` 文件，上线改连接串切换。
- uni-app 跨端，保留未来上 H5/App 的可能。
- NestJS 结构清晰，对单人长期维护友好。

**已知权衡**：
- SQLite 并发写入限制（同一时刻一个写）——家庭场景（几个人同时下单）完全够用。
- uni-app 跨端个别 API 各端有差异，但仅微信小程序 + H5 两端时坑很少。

### 2.4 关键设计取舍

1. **角色存在成员关系上**：支持一人多厨房多角色，不冗余、无变更冲突。
2. **主理人固定一人**：记录在 `Kitchen.owner_user_id`，转移需先指派新主理人。
3. **库存作为核心枢纽**：菜谱关联食材、订单完成扣库存、临期提醒、购物清单——所有联动绕库存转。
4. **订单可拒**：状态机含 `rejected`，拒单通知使用者。
5. **积分位预留**：菜品 `points` + 用户 `points_balance` 字段占位，MVP 不跑扣分逻辑。
6. **归属概念砍除**：库存为纯共享池，无 ownership 字段（费用/归属痛点归二期）。

## 3. 数据模型

### 3.1 关系总览

```
User ──< Membership >── Kitchen ──< 业务数据
                           ├──< Ingredient(食材字典)
                           ├──< Inventory(库存) >── Ingredient
                           ├──< Recipe(菜谱) ──< RecipeItem(用料) >── Ingredient
```

---

#### ⑨ Order 订单

| 字段 | 类型 | 说明 |
|---|---|---|
| id | PK | |
| kitchen_id | FK→Kitchen | |
| user_id | FK→User | 下单人（使用者） |
| status | enum | 状态机（见下） |
| total_points | int | 积分合计 **[预留，MVP=0]** |
| remark | string | 整单备注 |

**状态机**（订单可拒）：

```
pending ──主理人接单──> processing ──完成──> done (扣库存)
   │
   ├──主理人拒单──> rejected (通知使用者)
   │
   └──使用者撤单──> cancelled
```

- `pending`：刚下单，待处理
- `processing`：主理人接单，备餐中
- `done`：已完成，**触发自动扣库存**
- `rejected`：主理人拒单
- `cancelled`：使用者撤单（未接单前可撤自己单；主理人未接单前可撤任何单，接单后主理人可撤但需通知）

### 3.3 低库存阈值规则（修正后）

**采用方案 A：每个食材独立阈值，预填默认值。**

- `Ingredient.low_stock_threshold` 记录每种食材的阈值
- 录入食材时按 `unit` 预填默认值：`个→2`、`ml→200`、`g→20`、`kg→1`，主理人可改
- 购物清单查询：
  ```sql
  SELECT inventory.*, ingredient.name, ingredient.unit
  FROM inventory
  JOIN ingredient ON inventory.ingredient_id = ingredient.id
  WHERE inventory.kitchen_id = ?
    AND inventory.quantity <= ingredient.low_stock_threshold
  ```
- **修正记录**：原始"全局 ≤2"方案无法跨单位工作（ml/kg/g 无法用统一阈值），已修正为每食材独立阈值。

## 4. 角色权限矩阵

### 4.1 角色定义

| 角色 | 谁是 | 记录位置 | 一句话能力 |
|---|---|---|---|
| 主理人 owner | 固定一人 | `Kitchen.owner_user_id` + `Membership.role` | 管圈子的一切 + 处理订单 |
| 使用者 member | 其余成员 | `Membership.role='member'` | 下单 + 看自己的 + 看库存/购物清单 |

### 4.2 完整权限矩阵

R=可读 W=可写(增删改) X=执行

| 模块 | 操作 | 主理人 | 使用者 |
|---|---|:---:|:---:|
| **圈子 Kitchen** | 查看/切换厨房 | W | R（仅自己加入的） |
| | 改厨房名/邀请码 | W | — |
| | 邀请/踢人 | W | — |
| | 转移主理人 | W | — |
| **成员 Membership** | 看成员列表 | R | R |
| | 退出圈子 | X | X（owner 须先转） |
| **菜谱 Recipe** | 看菜谱库 | R（全部） | R（仅 `is_public=true`） |
| | 增删改菜谱 | W | — |
| **菜品 Dish** | 看菜单（上架的） | R | R |
| | 看下架菜品 | R | — |
| | 增删改/上下架 | W | — |
| **食材字典 Ingredient** | 看 | R | — |
| | 增删改 | W | — |
| **库存 Inventory** | 看库存 | R | **R（可看不能改）** |
| | 增删改库存 | W | — |
| | 看临期/低库存提醒 | R | R |
| **购物清单** | 看 | R | **R（可看不能改）** |
| | 标注"谁去买" | W | — |
| **订单 Order** | 下单 | X | **X（核心动作）** |
| | 看自己订单 | R | R（仅自己的） |
| | 看全部订单/看板 | R | — |
| | 接单/完成/拒单 | W | — |
| | 撤单（未接单前） | X（任何单） | X（仅自己的） |
| | 撤单（接单后） | X（需通知） | — |
| **通知** | 接收/查看 | R | R |

### 4.3 关键权限规则

1. 使用者能看库存（痛点④对所有成员有价值）、不能改。
2. 使用者能看公开菜谱（`is_public=true`），主理人可逐菜谱关闭公开。
3. 使用者只看自己订单，看不到别人的。
4. 使用者能看购物清单（可主动买菜），不能改。
5. 主理人固定一人，转移需先指派。

## 5. 模块流转

### 5.1 主链路：下单 → 自动扣库存

```
┌─────────────────────────────────────────────────────┐
│ 使用者端                                              │
│ ① 进菜单页(上架菜品) → 加购 → 购物车 → 提交订单      │
│    订单状态: pending                                  │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│ 主理人端（订单看板）                                   │
│ ② 看到 pending 订单 → 三选一:                        │
│    ├─ 接单 → processing                              │
│    ├─ 拒单 → rejected (通知使用者)                   │
│    └─ (不操作)                                       │
└──────────────────────┬──────────────────────────────┘
                       ▼ 接单后
┌─────────────────────────────────────────────────────┐
│ 主理人端                                              │
│ ③ 备餐完成 → 点"完成" → done                         │
│    触发自动扣库存                                     │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│ 系统自动扣减（决策 A② + B②）                          │
│ for each OrderItem:                                  │
│   if Dish.recipe_id 存在:                            │
│     for each RecipeItem in Recipe:                   │
│       消耗 = RecipeItem.quantity                     │
│              × OrderItem.quantity                    │
│              ÷ Recipe.servings                       │
│       按 ingredient_id 找该圈子匹配库存              │
│       按 expire_date 升序(先过期先扣) 扣减           │
│       不足 → 扣到0, 标记缺料, 通知主理人             │
│   else (Dish 无 recipe):                             │
│     不扣, 仅标记状态                                 │
└─────────────────────────────────────────────────────┘
```

**单位一致性提示**：按比例扣减（÷servings）要求 RecipeItem 用量与库存同单位同计量。录入菜谱用料时系统应给出单位一致性提示。

### 5.2 库存核心循环

```
        ┌──────────────────────────────────────────┐
        │                                          │
        ▼                                          │
   [入库] ───> [库存池] ───> [临期预警]             │
  (主理人)      Inventory      (自动)              │
                   │                │              │
        ┌──────────┼────────────────┼───┐          │
        ▼          ▼                ▼   │          │
   [订单完成    [手动消耗]    [过期清理] │          │
    自动扣减]   (主理人)     (主理人确认)│          │
        │          │                │   │          │
        └──────────┴────────────────┴───┘          │
                   │                               │
                   ▼                               │
            [购物清单] ≤阈值 ──────────────────────┘
            (使用者可见, 可主动买菜)   反向触发补货
```

**库存消耗 4 个出口**：订单自动扣减 / 手动消耗 / 过期清理 / 临期低库存→购物清单。

### 5.3 通知触发点

| 触发事件 | 接收人 | type |
|---|---|---|
| 订单被拒 | 下单使用者 | `order_rejected` |
| 订单接单 | 下单使用者 | `order_accepted` |
| 订单完成 | 下单使用者 | `order_done` |
| 订单缺料（扣减不足） | 主理人 | `stock_shortage` |
| 库存临期 | 全体成员 | `expiring_soon` |
| 库存过期 | 主理人 | `expired` |
| 接单后被主理人撤单 | 下单使用者 | `order_cancelled` |

MVP 实现：站内消息表 + 首页小红点（未读数）。

## 7. 关键业务规则汇总

1. **主理人固定一人**，记录于 `Kitchen.owner_user_id`，转移需先指派新主理人。
2. **角色记于 Membership**，支持一人多厨房多角色。
3. **订单可拒**，状态机含 `rejected`。
4. **自动扣减比例**：`RecipeItem.quantity × OrderItem.quantity ÷ Recipe.servings`。
5. **FIFO 扣减**：同 ingredient_id 多份库存按 `expire_date` 升序扣。
6. **无菜谱菜品不扣库存**：Dish.recipe_id 为空时订单完成仅标记状态。
7. **库存不足不阻断**：扣到 0、标记缺料、通知主理人，订单仍完成。
8. **低库存阈值每食材独立**：预填单位默认值，主理人可改。
9. **菜谱可逐个公开**：`is_public` 控制使用者可见性，默认 true。
10. **菜品积分位预留**：`points` 字段占位，MVP 不显示付款、不跑扣分。
11. **订单明细带快照**：`dish_snapshot` 保证历史订单不受菜品后续改动影响。
12. **单位一致性**：按比例扣减要求菜谱用料与库存同单位，录入时提示。

## 8. 决策记录（设计过程中的关键选择）

| # | 决策点 | 选择 | 理由 |
|---|---|---|---|
| D1 | MVP 范围 | 方案 A（库存为中心） | 直击第一梯队痛点，菜谱↔库存联动是产品灵魂 |
| D2 | 下单语义 | 报需求 + 结构化订单（购物车→下单） | 家庭场景最高频链路，C/D 模式放二期 |
| D3 | 付款环节 | 砍掉，预留积分位 | MVP 不做积分系统 |
| D4 | 订单可否拒单 | 可拒 | 家庭场景无人能强迫做饭 |
| D5 | 食材字典与库存是否合并 | 不合并 | 自动扣减需 ingredient_id 精确匹配 |
| D6 | 库存归属字段 | 砍除 | 归属痛点澄清为费用问题，归二期 |
| D7 | 扣减份数计算 | 精确按比例（÷servings） | 决策 A② |
| D8 | Dish 是否必须挂 Recipe | 不必须，无谱不扣 | 保留灵活性 |
| D9 | 菜谱公开性 | 逐菜谱 is_public 开关 | 比全局开关灵活 |
| D10 | 低库存阈值 | 每食材独立，预填默认 | 修正"全局≤2"无法跨单位的缺陷 |
| D11 | 通知 | 做轻量站内消息 | 家庭协调场景通知价值高 |
| D12 | 购物清单使用者可见 | 可见（只读） | 支持使用者主动买菜 |
